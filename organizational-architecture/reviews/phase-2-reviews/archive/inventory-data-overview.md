\# Inventory Dataset Schema  

\## Structured Dataset Blueprint



\---



\# Purpose



This document defines the structured dataset schema for the Northstar Health Operations Inventory \& Supply Operations subsystem.



The schema translates inventory activity into structured operational data that can later support:

\- CSV dataset creation

\- Excel analysis

\- SQL database construction

\- Power BI reporting

\- inventory KPI tracking

\- replenishment analysis

\- shortage monitoring

\- discrepancy analysis

\- vendor performance evaluation



\---



\# Dataset Design Overview



The Inventory \& Supply Operations subsystem will eventually use multiple related datasets rather than one single flat file.



This is intentional because inventory operations involve several different types of operational records:

\- inventory items

\- location stock levels

\- replenishment events

\- shortage events

\- inventory discrepancies

\- vendor shipment activity



This structure allows the subsystem to simulate realistic operational data relationships while remaining manageable for early-stage portfolio development.



\---



\# Planned Inventory Datasets



\## 1. inventory-items.csv



\### Dataset Grain



Each row represents:



```text

one inventory item

```



\### Purpose



This dataset defines the master list of inventory items tracked by Northstar Health Operations.



\### Key Fields



| Field Name | Data Type | Required | Description |

|---|---|---:|---|

| item\_id | Text | Yes | Unique inventory item identifier |

| item\_name | Text | Yes | Business-readable inventory item name |

| item\_category | Text | Yes | Operational supply category |

| criticality\_level | Text | Yes | Operational importance of item |

| unit\_of\_measure | Text | Yes | Measurement unit used for inventory tracking |

| preferred\_vendor\_id | Text | Yes | Primary vendor associated with item |

| active\_flag | Boolean | Yes | Indicates whether item is actively tracked |



\---



\## 2. location-inventory.csv



\### Dataset Grain



Each row represents:



```text

one item at one location

```



\### Purpose



This dataset tracks current stock levels, reorder points, and inventory status by operational location.



\### Key Fields



| Field Name | Data Type | Required | Description |

|---|---|---:|---|

| location\_inventory\_id | Text | Yes | Unique location inventory record identifier |

| item\_id | Text | Yes | Inventory item identifier |

| location\_id | Text | Yes | Clinic, hub, or operational location identifier |

| current\_stock | Numeric | Yes | Current stock quantity |

| reorder\_point | Numeric | Yes | Stock threshold that triggers replenishment review |

| target\_stock\_level | Numeric | Yes | Desired inventory level after replenishment |

| safety\_stock\_level | Numeric | Yes | Minimum buffer stock level |

| stock\_status | Text | Yes | Current inventory condition |

| last\_count\_date | Date | Yes | Most recent inventory count date |



\---



\## 3. replenishment-events.csv

Each row represents:



```text

one replenishment event

```



\### Purpose



This dataset tracks reorder activity, internal transfers, and replenishment workflow performance.



\### Key Fields



| Field Name | Data Type | Required | Description |

|---|---|---:|---|

| replenishment\_id | Text | Yes | Unique replenishment event identifier |

| item\_id | Text | Yes | Inventory item being replenished |

| location\_id | Text | Yes | Location receiving inventory |

| replenishment\_type | Text | Yes | Vendor reorder or internal transfer |

| requested\_quantity | Numeric | Yes | Quantity requested |

| approved\_quantity | Numeric | Conditional | Quantity approved |

| request\_date | Date | Yes | Date replenishment was requested |

| expected\_arrival\_date | Date | Conditional | Expected inventory arrival date |

| received\_date | Date | Conditional | Actual inventory receipt date |

| replenishment\_status | Text | Yes | Current replenishment workflow status |

| vendor\_id | Text | Conditional | Vendor involved if vendor reorder |

| related\_ticket\_id | Text | Optional | Ticket generated by related inventory issue |

\# Key Dataset Relationships



The inventory datasets are designed to connect through shared identifiers.



Important relationship fields include:

\- item\_id

\- location\_id

\- vendor\_id

\- related\_ticket\_id



Example relationship:



```text

Inventory shortage

↓

shortage-events.csv

↓

related\_ticket\_id

↓

tickets-v1.csv

↓

Ticketing \& Incident Management System

```



This creates cross-system analytical capability between inventory operations and ticketing workflows.
